# DDD (Domain Driven Design)

업무 규칙을 팀의 언어로 경계 안에 모아두는 것
- 업무 규칙 = 도메인 규칙 (Business Rules)
- 팀의 언어 = 개발자 + 기획자 + 현업이 “같은 뜻으로” 쓰는 말
- 업무 규칙은 특정한 경계(Bounded Context) 안에만 존재해야 한다

## Bounded Context
- **하나의 소프트웨어를 여러 모듈의 조합으로 나누는 것**
- 즉, 하나의 소프트웨어를 여러 **의미적 경계**로 나누는 것
- 각 경계는 독립적인 도메인 규칙과 모델을 가진다.
- 같은 용어라도 Bounded Context가 다르면 의미가 달라질 수 있다.
- 모듈은 최대한 독립적이고 폐쇄적으로 만든다.

## Ubiquitous Language
- **같은 개념에 이름 2개 이상 붙이지 않기**
- 예: 상품을 Goods, Product, Item 등으로 섞어서 부르지 않기
- 현업(실무담당자)이 사용하는 언어를 사용해야 한다.
- 현업에서 상품을 Product라고 부르면, 코드에서도 Product를 사용해야 한다. (Goods, Item 등 사용 금지)

## 규칙: 한 곳에 모으기
- **도메인 규칙은 한 곳에 모아둬야 한다.**
- 예: 상품 가격 할인 규칙
  - 상품 가격 할인 규칙이 여러 군데 흩어져 있으면 안 된다.
  - 상품 가격 할인 규칙이 바뀌면, 한 군데만 수정하면 된다.

## 비유
- ❌ DDD 없음 → 회사 규칙이 단톡방, 메일, 구두, 엑셀에 흩어져 있음
- ✅ DDD 있음 → “이 규칙은 인사팀 규정집 3장 2조”라고 딱 정해져 있음

## DDD의 목표
- 업무 변경이 발생했을 때 코드의 어느 부분을 바꿔야 하는지 바로 알 수 있는 구조를 만든다.

## 헥사고날 아키텍처 (Hexagonal Architecture)
- **애플리케이션의 중심에 도메인 로직을 두고, 외부와의 상호작용을 포트(port)와 어댑터(adapter)를 통해 처리하는 아키텍처 패턴**
- 포트(port): 도메인 로직이 외부와 상호작용하는 인터페이스
- 어댑터(adapter): 포트를 구현하여 외부 시스템과 도메인 로직 간의 통신을 담당하는 컴포넌트
- 헥사고날 아키텍처는 DDD의 도메인 중심 설계를 잘 실현하기 위한 대표적인 구조이다.

## MSA (Microservices Architecture)
- **애플리케이션을 독립적으로 배포 가능한 작은 서비스들로 나누는 아키텍처 스타일**
- 각 서비스는 특정 비즈니스 기능을 담당하며, 독립적으로 개발, 배포, 확장될 수 있다.
- MSA의 반대말은 모놀리식 아키텍처(Monolithic Architecture)이다.
- DDD와 MSA는 같은 개념이 아니다. DDD는 도메인 모델링과 설계에 중점을 두고, MSA는 애플리케이션의 아키텍처 스타일에 중점을 둔다.
- 그러나 DDD의 Bounded Context 개념은 MSA에서 각 마이크로서비스가 독립적인 도메인 모델을 가질 수 있도록 하는 데 도움이 된다.
